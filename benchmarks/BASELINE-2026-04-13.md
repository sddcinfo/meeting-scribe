## Root causes for today's "meeting crashes"

Not crashes ظ¤ three separate failure modes presenting as the same UX symptom:

1. **Auto-resume runs reprocess in parallel with a new meeting.**
   On scribe-main startup, `storage.recovered_meeting` sees the previously
   running meeting (the one that was alive when I killed scribe-main) and
   dev-mode auto-resumes it. The auto-resume triggers a 5-chunk chunked
   diarization pass over 33 min of audio. Each chunk runs synchronously
   enough to block the event loop ~2 s at a time. Your new meeting's
   audio ingestion starves; drift reached 41 s behind wall clock.
   - Evidence: log `Dev mode: auto-resuming interrupted meeting a1a028c8`
     at 15:08:55, followed by `Diarize chunk N/5` and `Event loop lag: 2.3s`
     interleaved with new-meeting audio chunks, until reprocess finished at
     15:10:35 and things settled.

2. **WebSocket disconnects after ~60ظô75 s without any apparent cause.**
   Every audio WS drops at roughly the same age. Server log shows:
   `WS connected (total=1)` ظْ 65 s later ظْ `WS error: Cannot call "receive"
   once a disconnect message has been received` ظْ `WS disconnected`. The
   client closes (browser side), not the server. uvicorn config at
   server.py:7209 doesn't set `ws_ping_interval` / `ws_ping_timeout` so
   whatever the default is applies. No server-level ping is being sent.
   - Hypothesis: WebSocket pings aren't keeping the connection alive under
     the CPU starvation from (1); browser's read buffer fills and the
     AudioWorklet pipeline decides to close.
   - Fix: set `ws_ping_interval=10`, `ws_ping_timeout=10` on both
     `admin_config` and `guest_config` at server.py:7209/7233.

3. **Legacy TTS 500 errors.** Every TTS synth from scribe-main for the
   meeting's interpretation-mode listeners returns a 500, so no audio
   fans out to browser listeners. Not a crash, but reinforces the
   "meeting appears broken" perception.

## Bandwidth contention caveat

Today's live meeting degradation chained back to **my earlier load test**
on vllm-tts (port 8022) with `concurrency=8, total=40`. That saturated
the GB10 LPDDR5x bus (273 GB/s shared across ASR + diarize + translate
+ TTS containers + vllm-tts spike). scribe-asr's vLLM engine watchdog
killed the worker. AudioHooks restarted, but in-flight audio replay is
what seeded the duplicate ASR finals I observed earlier.

**Rule going forward** (now saved as standing feedback):
do not run any container-level load tests while a meeting is live.
The GPU scheduler is not fair across processes; **vLLM priority headers
only order requests within a single vLLM instance**. Cross-container
bandwidth QoS needs cgroup / MPS, which we don't have.

## What "good" looks like (targets to defend)

Real-time meeting SLA:
- ASR first-final ظْ translate completion: **p50 ظëج 1.2 s, p95 ظëج 2.0 s**
- TTFA for TTS studio: **p50 ظëج 400 ms, p95 ظëج 600 ms** (new vllm-omni,
  c=1ظô2)
- utterance_end ظْ listener audio playout: **p50 ظëج 2 s, p95 ظëج 3 s**
- Event loop lag: **p95 < 200 ms**, **no > 1 s events while ingesting**
- Audio drift: **|drift| < 2 s steady-state**, resets between meetings
- WS reconnect: zero during an active meeting, unless client navigates

Current production (legacy stack): first three hold when no reprocess
contention + legacy TTS is replaced. The WS drop at ~60 s is the biggest
remaining outage source.

## Caveats

- `nvidia-smi` reports `[N/A]` for `memory.*` on the user account (GPU
  driver permissions); container-internal queries work. Fine for today.
- Translation and ASR latency was probed with synthetic tone audio. Real
  meeting audio is harder (speech + noise + short utterances). Expect
  real p50 to be **2ظô3 ├ù** these synthetic numbers.
- Event-loop-lag counts in the log are a blunt instrument ظ¤ they measure
  `asyncio.sleep(0) ظْ sleep returns` delta. Any spike > 100 ms is worth
  a note; > 1 s is always an incident.
