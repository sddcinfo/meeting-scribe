How It Works

仕組み

Real-time bilingual meeting transcription. Japanese and English, captured and translated side by side as the conversation flows.

会議の音声をリアルタイムで文字起こし・翻訳。日本語と英語を並べて表示し、会話の流れをそのまま記録します。

1

Audio Capture

音声キャプチャ

The browser captures audio from the system microphone using the Web Audio API. An AudioWorklet processor runs on a dedicated thread, resampling from the device's native rate (44.1 or 48 kHz) down to 16 kHz mono using linear interpolation.

ブラウザがWeb Audio APIでマイクの音声を取得します。AudioWorkletが専用スレッドで動作し、デバイスのネイティブレート（44.1/48 kHz）から16 kHz モノラルに線形補間でリサンプリングします。

Every 250 milliseconds, a chunk of approximately 4,000 samples is packaged with a binary header and sent over a WebSocket connection. No JSON encoding, no base64 — pure binary PCM for minimum latency.

250ミリ秒ごとに約4,000サンプルをバイナリヘッダと共にWebSocketで送信します。JSONもbase64も使いません。純粋なバイナリPCMで最小レイテンシを実現します。

Microphone→ AudioWorklet→ Resample 16kHz→ WebSocket→ Server
// Wire format — binary WebSocket frame [4B uint32] sample_rate // 16000 [8B uint64] sample_offset // absolute position [4B uint32] stream_epoch // reconnection counter [... ] Float32 PCM // ~4000 samples
Why client-side resampling?
なぜクライアント側でリサンプリング？
Offloading resampling to the browser means zero audio processing overhead on the server. The server receives clean 16 kHz PCM and feeds it directly to the ASR model — no conversion step, no buffering delay. リサンプリングをブラウザに任せることで、サーバー側の音声処理負荷がゼロになります。サーバーはクリーンな16 kHz PCMをそのままASRモデルに渡すだけ。変換も遅延もありません。
2

Speech Recognition

音声認識

Incoming audio is accumulated in a 4-second buffer. When voice activity is detected (RMS energy above threshold), the buffer is transcribed by the ASR backend. The system auto-detects Japanese and English, producing timestamped transcript segments.

受信した音声は4秒のバッファに蓄積されます。音声活動が検出される（RMSエネルギーが閾値を超える）と、ASRバックエンドが文字起こしを実行。日本語と英語を自動判別し、タイムスタンプ付きセグメントを生成します。

The ASR backend is pluggable — on Apple Silicon, Qwen3-ASR-0.6B runs natively via MLX with a real-time factor as low as 0.06 (16x faster than real-time). On NVIDIA GPUs, faster-whisper with the large-v3-turbo model provides higher quality with CUDA INT8 acceleration.

ASRバックエンドは差し替え可能です。Apple SiliconではQwen3-ASR-0.6BがMLXでネイティブに動作し、リアルタイムファクター0.06（実時間の16倍速）を達成。NVIDIA GPUではfaster-whisperのlarge-v3-turboモデルがCUDA INT8で高精度な認識を提供します。

Hallucination filtering
ハルシネーション対策
ASR models sometimes hallucinate during silence. Meeting Scribe applies multi-layer filtering: known phrases are blocked, segments over 200 characters are discarded, and text with 50%+ repeating characters is rejected. Language identification uses CJK character ratio heuristics. ASRモデルは無音区間でハルシネーションを起こすことがあります。Meeting Scribeは多層フィルタで対策：既知フレーズのブロック、200文字超のセグメント除去、50%以上の繰り返し文字除去。言語判別にはCJK文字比率を使用します。
4s buffer→ VAD→ ASR model→ Filter→ TranscriptEvent
3

Translation Pipeline

翻訳パイプライン

Finalized transcript segments enter the translation pipeline. A merge gate holds segments briefly and combines adjacent utterances in the same language (within a 2-second gap from the same speaker) — reducing translation overhead for short, fragmented speech.

確定したセグメントが翻訳パイプラインに入ります。マージゲートがセグメントを一時保持し、同一言語・同一話者の隣接発話（2秒以内）を結合。短い断片的な発話のオーバーヘッドを削減します。

Merged segments are placed in an async queue with 2 concurrent workers, a capacity of 50 items, and a 30-second timeout per translation. When full, the oldest unstarted item is dropped — the system never falls behind.

結合されたセグメントは非同期キューへ。2つの並行ワーカー、最大50アイテム、翻訳あたり30秒のタイムアウト。満杯時は未着手の最古アイテムを破棄し、リアルタイム性を維持します。

TranscriptEvent→ Merge Gate→ Async Queue→ Backend→ Display

The default backend is NLLB-200 (1.3B parameters) via CTranslate2 with INT8 quantization. CPU: ~500ms/sentence. CUDA: ~100ms/sentence. For higher quality, vLLM + Qwen3.5 serves as an alternative.

4

Speaker Identification

話者識別

Before a meeting begins, participants can enroll their voice by speaking for 2–3 seconds. The system extracts a voiceprint embedding using ECAPA-TDNN (SpeechBrain), a compact neural network trained on speaker verification.

会議の開始前に、参加者は2〜3秒の発話で声を登録できます。ECAPA-TDNN（SpeechBrain）で声紋の埋め込みベクトルを抽出。話者照合に特化したコンパクトなニューラルネットワークです。

During the meeting, diart (powered by pyannote) performs online speaker diarization — clustering audio into speaker turns in real time, supporting up to 4 concurrent speakers. Each cluster is matched against enrolled embeddings via cosine similarity.

会議中はdiart（pyannoteベース）がオンラインで話者ダイアライゼーションを実行。音声をリアルタイムで話者ごとにクラスタリングし、最大4人に対応。コサイン類似度で登録済み埋め込みと照合して名前を割り当てます。

Enrollment→ ECAPA-TDNN→ Voiceprint
Live audio→ diart→ Cluster match→ Named speaker
Optional by design
任意の機能として設計
Speaker identification is entirely optional. Meetings can start without enrollment — speakers get cluster labels (Speaker 1, 2) based on voice similarity. Names can be added anytime. 話者識別は完全にオプションです。登録なしで会議を開始でき、声の類似性に基づいてクラスタラベル（話者1、話者2）が自動割り当て。名前はいつでも追加できます。
5

The Display

表示

The frontend renders a 3-column synchronized grid using CSS Grid. Each segment spans all three columns as a single row, ensuring perfect vertical alignment. Live shows all speech as-spoken; English and Japanese columns show originals and translations.

フロントエンドはCSS Gridによる3カラム同期グリッドで表示。各セグメントが3カラムにまたがる1行として配置され、縦方向の位置が完全に揃います。ライブ列は全発話を原語で表示し、英語・日本語列にはそれぞれの原文と翻訳が表示されます。

New segments enter with a slide-up animation. Translations flash with a gold highlight on arrival. Speaker attribution uses color-coded badges — consistent colors throughout the meeting.

新しいセグメントはスライドアップで表示。翻訳到着時はゴールドのハイライトで一瞬光ります。話者はカラーバッジで表示され、会議を通じて一貫した色が使われます。

The entire frontend is built with zero build tools — static HTML, CSS, and vanilla JavaScript. No bundler, no transpiler, no framework.

フロントエンド全体がビルドツールなしで構築。静的HTML、CSS、バニラJavaScriptのみ。バンドラーもフレームワークもありません。
