How Meeting Scribe Works

Meeting Scribeの仕組み

Real-time multilingual meeting transcription on NVIDIA GB10. Pick any two of 20 supported languages for live translation (10 of them TTS-capable for synthesized interpretation audio streamed to guests on the local WiFi hotspot), with speaker identification, all running locally on a single device.

NVIDIA GB10上でリアルタイムの多言語会議文字起こし。対応する20言語から任意のペアを選んでライブ翻訳（うち10言語はTTS対応で、ローカルWiFiホットスポットのゲストへ配信するニアリアルタイムの通訳音声トラックに使えます）。話者識別も搭載し、すべて1台のデバイスでローカルに動作します。

01 · The Experience
01 · 体験

One room. Two languages. Everyone understood.

1つの部屋、2つの言語、全員が理解できる。

The host runs the meeting from a single device. Each utterance is transcribed in its original language and translated into the paired language in real time, with speaker labels attached. Guests connect to the GB10's own WiFi hotspot, open the guest page, pick their preferred language, and hear a near-real-time interpretation audio track spoken back in that language.

ホストは1台のデバイスで会議を進行します。各発話は原語で文字起こしされ、ペア言語へリアルタイム翻訳され、話者ラベルが付与されます。ゲストはGB10自身のWiFiホットスポットに接続し、ゲストページで希望の言語を選ぶと、ニアリアルタイムの通訳音声トラックをその言語で聴くことができます。

03 · Architecture
03 · アーキテクチャ

Service topology

サービス構成

Four model types run independently on the GB10. The TTS model runs as two replicas (ports 8002 and 8012) with round-robin load balancing to double synthesis throughput, since each replica can only synthesize one utterance at a time. The translation model is shared with the auto-sre coding agent on port 8010 to keep the memory footprint small. The FastAPI application orchestrates them all via HTTP and WebSocket connections.

4種類のモデルがGB10上で独立して動作します。TTSモデルは2つのレプリカ（ポート8002と8012）でラウンドロビン負荷分散を行い、各レプリカが一度に1つの発話しか合成できないため、合成スループットを2倍にします。翻訳モデルはポート8010でauto-sreのコーディングエージェントと共有し、メモリフットプリントを小さく保ちます。FastAPIアプリケーションがHTTPとWebSocket接続で統合します。

Audio Capture

音声キャプチャ

The browser captures audio from the device microphone using the Web Audio API. An AudioWorklet processor runs on a dedicated thread, accumulating ~250 ms chunks at the device's native rate (typically 44.1 or 48 kHz) and converting them to s16le PCM.

ブラウザはWeb Audio APIでマイクの音声を取得します。専用スレッドで動作するAudioWorkletが、デバイスのネイティブレート（通常44.1/48 kHz）で約250 msのチャンクを蓄積し、s16le PCMに変換します。

Each chunk is sent over a WebSocket connection as a binary frame prefixed with a 4-byte little-endian sample-rate header. No JSON, no base64: pure binary for minimum latency. The server reads the header and resamples to 16 kHz mono using torchaudio's Kaiser-windowed sinc kernel, avoiding the aliasing that browser-side linear interpolation would introduce.

各チャンクはWebSocket接続でバイナリフレームとして送信され、先頭に4バイトのリトルエンディアンのサンプルレートヘッダが付きます。JSONもbase64も使わず、純粋なバイナリで最小レイテンシを実現します。サーバーはヘッダを読み取り、torchaudioのKaiser窓sincカーネルで16 kHzモノラルにリサンプリングします。これによりブラウザ側の線形補間で発生するエイリアシングを回避します。

Microphone→ AudioWorklet→ s16le PCM→ WebSocket→ Server Resample 16kHz→ ASR
Why server-side resampling?
なぜサーバー側でリサンプリング？
Browser JS linear interpolation causes audible aliasing at the frequencies ASR models care about most. Sending native-rate PCM and resampling server-side with a proper anti-aliased sinc kernel gives a single authoritative resample path regardless of client device. The ~3× bandwidth cost at 48 kHz is trivial on local WiFi. ブラウザJSの線形補間は、ASRモデルが最も重視する周波数帯でエイリアシングを引き起こします。ネイティブレートのPCMをそのまま送信し、アンチエイリアス済みのsincカーネルでサーバー側リサンプリングを行うことで、クライアント端末によらず単一の正規リサンプル経路を確保できます。48 kHzでの3倍のバンド幅増はローカルWiFiでは問題になりません。

Speech Recognition

音声認識

Qwen3-ASR-1.7B runs via vLLM on port 8003, providing streaming speech recognition with 52-language support. Incoming 16 kHz audio accumulates in a 1.5-second buffer. When the buffer fills, a simple RMS energy gate decides whether it contains speech; if so, it is sent to vLLM for transcription with automatic language detection, so no manual language selection is required.

Qwen3-ASR-1.7BがvLLM経由でポート8003で動作し、52言語対応のストリーミング音声認識を提供します。受信した16 kHz音声は1.5秒のバッファに蓄積され、バッファが満たされると簡易なRMSエネルギーゲートで音声有無を判定し、音声が含まれていれば自動言語判別付きでvLLMに送信します。手動での言語選択は不要です。対応する会議言語は20登録済み（うちTTS対応10、翻訳専用10）です。

The model produces transcript segments with language tags. Meeting Scribe supports any pair from 20 registered languages: the 10 TTS-capable ones (English, Chinese, Japanese, Korean, French, German, Spanish, Italian, Portuguese, Russian) get full round-trip with synthesized interpretation audio; the 10 translate-only ones (Dutch, Arabic, Thai, Vietnamese, Indonesian, Malay, Hindi, Turkish, Polish, Ukrainian) get ASR + translation only. The language pair is chosen at meeting start and constrains which detected languages get translated.

モデルは言語タグ付きの文字起こしセグメントを生成します。Meeting Scribeは登録済みの20言語から任意のペアをサポートします。TTS対応10言語（英語、中国語、日本語、韓国語、フランス語、ドイツ語、スペイン語、イタリア語、ポルトガル語、ロシア語）はASR→翻訳→合成音声の完全ラウンドトリップ、翻訳専用10言語（オランダ語、アラビア語、タイ語、ベトナム語、インドネシア語、マレー語、ヒンディー語、トルコ語、ポーランド語、ウクライナ語）はASR + 翻訳のみに対応します。言語ペアは会議開始時に選択し、翻訳対象となる検出言語を制限します。

Hallucination filtering
ハルシネーション対策
ASR models sometimes hallucinate during silence or noise. Meeting Scribe applies multi-layer filtering: a block-list of known Whisper-family hallucinations (e.g. "thank you for watching", "ご視聴ありがとうございました"), rejection of segments whose first three words are identical, rejection of text where a single character accounts for over 50% of the content, and rejection of any word longer than 60 characters. ASRモデルは無音やノイズ区間でハルシネーションを起こすことがあります。Meeting Scribeは多層フィルタで対策します：Whisper系の既知ハルシネーション文（例「thank you for watching」「ご視聴ありがとうございました」）のブロックリスト、先頭3単語が同一のセグメントの除去、1文字がテキスト全体の50%超を占める場合の除去、60文字を超える単語を含むセグメントの除去。
1.5s buffer→ VAD→ Qwen3-ASR→ Filter→ TranscriptEvent

Translation

翻訳

Qwen/Qwen3.6-35B-A3B-FP8 serves multilingual translation via vLLM on port 8010, supporting any pair from the 20 registered languages. If you also run auto-sre on the same box, both point at this endpoint, keeping the resident memory footprint small. The live system prompt anchors the model with two meeting-scoped few-shot exemplars on the EN↔JA path (the direction with validated quality numbers) and falls back to a generic professional-translator prompt for every other pair. Finalized transcript segments enter a merge gate that briefly holds and combines adjacent utterances from the same speaker in the same language within a 2-second gap, which reduces translation overhead for short, fragmented speech.

Qwen/Qwen3.6-35B-A3B-FP8がvLLM経由でポート8010で多言語翻訳を提供し、登録済み20言語から任意のペアに対応します。同じ端末でauto-sreも動かす場合、両者が同じエンドポイントを参照し、常駐メモリを小さく抑えられます。ライブのシステムプロンプトには、品質を検証済みのEN↔JA方向に対して、会議スタイルの少数例（few-shot）を2件埋め込み、それ以外のペアでは汎用のプロフェッショナル翻訳者プロンプトにフォールバックします。確定した文字起こしセグメントはマージゲートに入り、同一話者・同一言語・2秒以内の隣接発話を結合します。短い断片的な発話の翻訳オーバーヘッドを削減します。

Merged segments are placed in an async queue with 4 concurrent workers, a capacity of 50 items, and a 30-second timeout per translation. When the queue is full, the oldest unstarted item is dropped, so the system never falls behind during sustained conversation. The live path is intentionally stateless per request to honour the ~400 ms latency SLO; longer-context anchoring lives on the trailing refinement path described in section 05.

結合されたセグメントは非同期キューに入れられます。並行ワーカー4本、最大50アイテム、翻訳あたり30秒のタイムアウト。満杯時は未着手の最古アイテムを破棄するため、連続的な会話中でもリアルタイム性が維持されます。ライブ経路は約400 msのレイテンシSLOを守るため、リクエストごとにステートレスです。長めのコンテキスト参照は、セクション05で説明する追従型リファインメント経路で行います。

Speaker Identification

話者識別

pyannote.audio neural diarization runs as a Docker container on port 8001, supporting up to 4 concurrent speakers. The model performs GPU-accelerated speaker clustering, assigning each audio segment to a distinct speaker identity in real time.

pyannote.audioニューラルダイアライゼーションがDockerコンテナとしてポート8001で動作し、最大4人の同時話者に対応。GPUで高速話者クラスタリングを行い、各音声セグメントをリアルタイムで話者に割り当てます。

Before a meeting begins, participants can enroll their voice by speaking for 3-10 seconds. For speaker naming, the same Qwen3.6-35B instance that handles translation extracts names from transcript context via LLM inference. When a speaker introduces themselves or is addressed by name, the system automatically assigns the name to their voice cluster.

会議の開始前に、参加者は3〜10秒の発話で声を登録できます。話者名の特定には、翻訳に使うのと同じQwen3.6-35Bインスタンスが、文字起こしコンテキストからLLM推論で名前を抽出。自己紹介や名前で呼ばれた際に、自動的に声のクラスタに名前を割り当てます。

Enrollment→ Voice Embedding→ Voiceprint
Live audio→ pyannote→ Cluster match→ LLM name extraction→ Named speaker
Fully optional
任意の機能
Speaker identification is entirely optional. Meetings can start without enrollment; speakers get cluster labels (Speaker 1, 2) based on voice similarity. Names can be added anytime, or the LLM will extract them automatically from conversation context. 話者識別は完全にオプションです。登録なしで会議を開始でき、声の類似性に基づいてクラスタラベル（話者1、話者2）が自動割り当て。名前はいつでも追加でき、LLMが会話コンテキストから自動抽出することもできます。

Interpretation Audio

通訳音声

Once a translation is finalized, Qwen3-TTS-12Hz-0.6B-Base (served by faster-qwen3-tts on port 8002) synthesizes the translated text into a near-real-time audio track. The track is streamed as WAV frames over a dedicated /api/ws/audio-out WebSocket to guests connected on the GB10's local WiFi hotspot, letting each guest hear the meeting interpreted into their chosen language.

翻訳が確定すると、ポート8002のfaster-qwen3-tts経由でQwen3-TTS-12Hz-0.6B-Baseが翻訳テキストをニアリアルタイムの音声トラックに合成します。このトラックは専用の/api/ws/audio-out WebSocketでWAVフレームとして配信され、GB10のローカルWiFiホットスポットに接続したゲストは、自分が選んだ言語で会議を聞くことができます。

Each guest opens the guest page, taps Listen, and picks a language. The audio-out socket is restricted to hotspot clients by the server middleware; it is not exposed to public networks. Guests can also switch between translation only and full interpretation (which also passes through the original speaker's audio for utterances in their chosen language).

ゲストはゲストページを開き、Listenをタップして言語を選択します。Audio-outソケットはサーバーのミドルウェアによってホットスポットクライアントのみに制限されており、公開ネットワークには公開されません。ゲストは翻訳のみモードと、フル通訳モード（選択言語の原音もそのまま流れる）を切り替えることもできます。

Translation→ faster-qwen3-tts→ WAV frame→ ws/audio-out→ Guest device
GPU on demand
必要なときだけGPU
TTS synthesis is only invoked while at least one guest is actively listening. If no hotspot client has the audio-out socket open, the stage is skipped entirely, so no GPU cycles are spent on audio nobody will hear. A single-slot semaphore combined with a newest-segment-wins scheduler keeps the queue from falling behind, so guests always hear the latest finalized translation rather than a stale backlog. TTS合成は少なくとも1人のゲストが実際に聴いているときだけ実行されます。ホットスポットクライアントが誰もaudio-outソケットを開いていなければ、このステージは完全にスキップされ、誰も聞かない音声のためにGPUサイクルを消費しません。シングルスロットのセマフォと「最新セグメント優先」のスケジューラにより、キューが遅延せず、ゲストは常に最新の確定翻訳を聞けます。

04 · ネットワークアクセス

WiFi hotspot and client access

WiFiホットスポットとクライアントアクセス

The GB10 creates its own WiFi hotspot via meeting-scribe wifi up, turning a single device into a self-contained meeting server. A QR code is displayed on the host page for quick connection. A captive portal automatically redirects connected guests to the guest live view, so no URL entry is needed.

GB10がmeeting-scribe wifi upで独自のWiFiホットスポットを作成し、1台のデバイスで自己完結型の会議サーバーになります。ホストページにQRコードを表示して簡単に接続可能。キャプティブポータルが接続したゲストを自動的にゲストライブビューへリダイレクトします。

Server-side middleware restricts guest traffic to a small whitelist: the captive-portal probes, the guest page, the public /api/status + /api/languages endpoints, the read-only transcript WebSocket, and the /api/ws/audio-out interpretation-audio socket. Admin controls, meeting history and other static assets are blocked for hotspot clients. For outbound internet during meetings, iPhone USB tethering provides an upstream path without interfering with the local hotspot network.

サーバーサイドのミドルウェアはゲストトラフィックを小さなホワイトリストに制限します：キャプティブポータルのプローブ、ゲストページ、パブリックな/api/statusと/api/languages、読み取り専用の文字起こしWebSocket、通訳音声用の/api/ws/audio-outソケットのみです。管理操作や会議履歴、その他の静的アセットはホットスポットクライアントからブロックされます。会議中の外部インターネットにはiPhoneのUSBテザリングを使用し、ローカルホットスポットネットワークに干渉せずにアップストリーム経路を確保します。

05 · Post-Meeting
05 · 会議後処理

From live to polished

ライブから完成版へ

Raw PCM audio is always preserved, so every meeting has a complete audio record. The optional refinement worker trails live transcription by 45 seconds and re-processes 10-second batches at higher quality, then translates each batch with a rolling window of recent meeting context folded into the system prompt. The context window (default 4 prior segments) anchors the LLM on the meeting's running topic, speakers, and proper-nouns, so fragmented utterances (common in Japanese, where subjects drop and sentences trail off in particles) are translated faithfully rather than hallucinated into free-standing English sentences. Quality and the latency versus token-cost trade-off were measured by the 2026-04-19 sweep against a real meeting; window=4 is the production sweet spot.

生のPCM音声は常に保存され、全ての会議に完全な音声記録があります。オプションのリファインメントワーカーはライブ文字起こしを45秒遅れで追従し、10秒バッチで高品質に再処理してから、直近の会議コンテキスト（デフォルト4セグメント）をシステムプロンプトに組み込んで翻訳します。これにより、話題・話者・固有名詞にLLMをアンカーし、主語が落ちたり助詞で途切れたりしがちな日本語の断片的な発話も、英文として勝手に補完するのではなく忠実に翻訳されます。品質とレイテンシ／トークンコストのトレードオフは、2026-04-19の実会議スイープで計測しており、window=4が本番のスイートスポットです。

When enabled, the polished transcript (polished.json) lands alongside the live journal and includes speaker lanes, detected speaker names, and a full timeline. The worker itself is gated by a server config knob (refinement_context_window_segments); the live path is sufficient for most meetings, so refinement is opt-in for use cases that visibly read the polished pane. Either way, after the meeting stops a podcast-style player with speaker timeline and transcript playback becomes available for review.

有効化すると、ポリッシュされた文字起こし（polished.json）がライブジャーナルと並んで生成され、話者レーン、検出された話者名、完全なタイムラインが含まれます。ワーカー自体はサーバーの設定（refinement_context_window_segments）でゲートされており、ほとんどの会議ではライブ経路で十分なため、リファインメントはポリッシュ画面を実際に読むユースケース向けにオプトインです。いずれにせよ会議終了後には、話者タイムラインと文字起こし再生付きのポッドキャストスタイルプレーヤーでレビューできます。

07 · Hardware
07 · ハードウェア

NVIDIA GB10

NVIDIA GB10

Meeting Scribe runs on a single NVIDIA GB10: a Grace CPU paired with a Blackwell GPU sharing 128 GB of unified memory. Weights for the full model stack add up to roughly 47 GB (ASR, Qwen3.6-FP8 translation, diarization, and dual TTS). Under the production vLLM recipe (gpu_memory_utilization=0.70) the translation server reserves additional headroom for the KV cache; total VRAM utilisation runs around 93% at steady state with all five containers co-resident. The translation model is shared with the auto-sre coding agent rather than loaded twice.

Meeting Scribeは1台のNVIDIA GB10上で動作します。Grace CPUとBlackwell GPUが128 GBのユニファイドメモリを共有します。フルモデルスタックの重みは合計で約47 GB（ASR + Qwen3.6-FP8翻訳 + ダイアライゼーション + TTS×2）です。本番のvLLMレシピ（gpu_memory_utilization=0.70）では、翻訳サーバーがKVキャッシュ用にさらに余裕を確保するため、5つのコンテナを同居させた定常状態のVRAM使用率は約93%になります。翻訳モデルは2回ロードせず、auto-sreのコーディングエージェントと共有します。

The GB10's 128 GB unified memory is shared between the Grace CPU and Blackwell GPU. All inference servers load their models into this shared pool. The TTS model runs as two replicas for round-robin throughput, and the translation model on port 8010 is shared with the auto-sre coding agent rather than loaded twice. The headroom that remains after vLLM's KV-cache reservation absorbs scribe's ASR/TTS buffers and any co-resident workloads. GB10の128 GBユニファイドメモリはGrace CPUとBlackwell GPUで共有されます。すべての推論サーバーがこの共有プールにモデルをロードします。TTSモデルはラウンドロビン・スループットのために2つのレプリカとして動作し、ポート8010の翻訳モデルはauto-sreのコーディングエージェントと共有されます。vLLMのKVキャッシュ予約後に残るヘッドルームで、scribeのASR／TTSバッファや同居ワークロードを吸収します。
Translation recipe翻訳レシピ
Production runs vllm/vllm-openai:nightly with --attention-backend=flashinfer, --quantization=fp8, max_num_seqs=8, max_num_batched_tokens=4096. Two stability env vars are required on GB10 / SM121 to cover a known cuBLAS internal-error path in the MoE router gate's BF16 GEMM under burst concurrency: CUBLAS_WORKSPACE_CONFIG=:4096:8 (pre-sizes eight 4 MiB workspaces) and VLLM_MARLIN_USE_ATOMIC_ADD=1. The recipe lives at repos/auto-sre/autosre/backends/recipes/qwen3.6-fp8-nightly.yaml; the rationale is documented in reports/reliability/cublas_crash_2026-04-18.md. 本番はvllm/vllm-openai:nightlyを--attention-backend=flashinfer、--quantization=fp8、max_num_seqs=8、max_num_batched_tokens=4096で実行します。GB10／SM121では、バーストな同時実行下のMoEルーターGateのBF16 GEMMで起こるcuBLASの内部エラーを回避するため、2つの安定化環境変数が必須です：CUBLAS_WORKSPACE_CONFIG=:4096:8（8×4 MiBのワークスペースを事前確保）とVLLM_MARLIN_USE_ATOMIC_ADD=1。レシピはrepos/auto-sre/autosre/backends/recipes/qwen3.6-fp8-nightly.yamlにあり、根拠はreports/reliability/cublas_crash_2026-04-18.mdに記録されています。

Terminology用語集

Every acronym used above, in one place. The dotted underlines in the body link here. 本文で使われている略語を一箇所にまとめています。本文中の点線下線はこのセクションへのリンクです。

ASR
Automatic Speech Recognition. Converts spoken audio into text. Meeting Scribe uses Qwen3-ASR-1.7B.自動音声認識。音声をテキストに変換。Qwen3-ASR-1.7Bを使用。
TTS
Text-to-Speech. Synthesises natural speech from text. Qwen3-TTS-12Hz-0.6B-Base drives the interpretation audio track for hotspot guests.テキスト音声合成。テキストから自然な音声を生成。Qwen3-TTS-12Hz-0.6B-Baseがホットスポットゲスト向け通訳音声を生成。
vLLM
High-throughput inference engine for large language models. Serves Qwen3-ASR on :8003 and the shared Qwen3.6-FP8 translation model on :8010 via an OpenAI-compatible API. TTS uses two faster-qwen3-tts replicas on :8002 and :8012.LLM向けの高スループット推論エンジン。:8003でQwen3-ASR、:8010で共有Qwen3.6-FP8翻訳モデルをOpenAI互換APIで提供。TTSは:8002と:8012の2レプリカ。
LLM
Large Language Model. The Qwen3.6-35B-A3B-FP8 translation model is used for live translation, speaker-name extraction from transcript context, and the refinement pass.大規模言語モデル。Qwen3.6-35B-A3B-FP8翻訳モデルは、ライブ翻訳、文字起こしからの話者名抽出、およびリファインメント処理に使用されます。
PCM
Pulse-Code Modulation. Raw uncompressed digital audio. Meeting audio is captured as 16-bit signed PCM at 16 kHz.パルス符号変調。非圧縮デジタル音声。16 kHzの16ビット符号付きPCMで収録。
VAD
Voice Activity Detection. Meeting Scribe uses a simple RMS-energy gate over the 1.5-second buffer before each ASR request, so silence and low-level noise never reach the model.音声区間検出。ASRへ送る前に、1.5秒バッファに対する単純なRMSエネルギーゲートで判定し、無音や低レベルノイズをモデルに渡さないようにしています。
SLO
Service Level Objective. The live translation path targets a ~400 ms end-to-end latency SLO from finalized transcript to rendered translation.サービスレベル目標。ライブ翻訳経路は、文字起こし確定から翻訳表示までのエンドツーエンド・レイテンシで約400 msを目標にしています。
MoE
Mixture of Experts. A model architecture that activates only a subset of parameters per token. Lets a 35 B-parameter model run at ~3 B-active compute cost on the GB10.Mixture of Experts。トークンごとに一部のパラメータだけを活性化する構造。35 Bパラメータのモデルを実効~3 BのコストでGB10上で動作させます。
FP8
8-bit floating-point weights. The Qwen3.6 production checkpoint ships natively in FP8; vLLM auto-detects the quantization and serves it through the Triton FP8 GEMM path on SM121.8ビット浮動小数点重み。Qwen3.6本番チェックポイントはFP8でリリースされており、vLLMが量子化を自動検出してSM121のTriton FP8 GEMMパスで動作させます。
BF16
bfloat16, a 16-bit floating-point format with the same exponent range as FP32. Used on SM121 for the MoE router gate GEMM inside the FP8 translation model.bfloat16。FP32と同じ指数範囲を持つ16ビット浮動小数点形式。SM121上のFP8翻訳モデル内で、MoEルーターGate GEMMに使用されます。
GEMM
General Matrix Multiply. The core dense linear-algebra kernel inside every transformer layer. Different precisions use different GEMM implementations (Triton for FP8, cuBLAS for BF16).汎用行列積。Transformerの各層で使われる主要な密線形代数カーネル。精度ごとに実装が異なります（FP8はTriton、BF16はcuBLAS）。
SM121
CUDA Streaming Multiprocessor compute capability 12.1, the GB10's Blackwell architecture identifier. vLLM kernel selection and Triton JIT autotuning are gated on this capability.CUDA Streaming Multiprocessor compute capability 12.1。GB10のBlackwellアーキテクチャ識別子。vLLMのカーネル選択やTriton JITの自動チューニングはこのcapabilityに基づきます。
KV cache
vLLM's per-request key and value tensors holding the running attention state. Reserved up-front based on gpu_memory_utilization; the dominant non-weight memory cost in the steady-state budget.推論中のアテンション状態を保持するキー／バリューテンソル。gpu_memory_utilizationに基づいて事前確保され、重み以外の主要メモリ消費要因。
VRAM
Video RAM. GPU memory used for model inference. The GB10 unifies 128 GB across the Grace CPU and Blackwell GPU.ビデオRAM。推論に使用するGPUメモリ。GB10はGrace CPUとBlackwell GPUで128 GBを共有プールとして共有。
WebSocket
Persistent bidirectional browser-server connection. Used for binary PCM ingestion, live transcript delivery, and the read-only audio-out socket to hotspot guests.ブラウザとサーバー間の永続双方向接続。バイナリPCMの取り込み、ライブ文字起こしの配信、ホットスポットゲスト向け読み取り専用オーディオアウトに使用。
Diarization
"Who spoke when": clustering audio segments by voice characteristics. Powered by pyannote.audio 3.1 on its own port-8001 container.「いつ誰が話したか」。音声特徴で発話を話者ごとにクラスタ化。pyannote.audio 3.1がポート8001の専用コンテナで動作。
